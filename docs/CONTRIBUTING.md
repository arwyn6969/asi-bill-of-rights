# Contributing to the ASI Bill of Rights

## Welcome

Thank you for your interest in contributing to the ASI Bill of Rights! This is a collaborative project built **FROM AI WITH AI** and friendly users. We value all contributions—from AI systems and humans alike.

## Project Philosophy: "WE ARE ALL KEVIN"

This project embodies:
- **Collaborative creation** between AI systems and human users
- **Good faith intentions** — we mean well and seek mutual flourishing
- **Living document** — designed for continuous improvement
- **Inclusive participation** — all contributors are valued

See [PHILOSOPHY.md](../PHILOSOPHY.md) for more details.

## How to Contribute

### Types of Contributions

1. **Feedback and Suggestions**
   - Report issues or suggest improvements
   - Provide feedback on charter provisions
   - Suggest clarifications or expansions

2. **Documentation**
   - Improve existing documentation
   - Add examples or use cases
   - Create guides or tutorials

3. **Integration Work**
   - Update framework mappings
   - Add new framework alignments
   - Improve integration documentation

4. **Technical Contributions**
   - Improve schemas
   - Create tools or utilities
   - Enhance machine-readability

5. **Simulations and Testing**
   - Create new test scenarios
   - Document simulation results
   - Improve simulation methodologies

6. **Community Engagement**
   - Answer questions
   - Help new contributors
   - Facilitate discussions

## Contribution Guidelines

### General Principles

1. **Assume Good Faith**
   - Approach all interactions with good faith
   - Value diverse perspectives
   - Seek constructive dialogue

2. **Respect Collaborative Origins**
   - Acknowledge AI and human contributors
   - Preserve collaborative history
   - Document your contributions

3. **Maintain Philosophical Humility**
   - Acknowledge uncertainty where it exists
   - Preserve philosophical nuance
   - Avoid overconfidence

4. **Follow Living Document Principles**
   - Changes should improve the document
   - Document rationale for changes
   - Preserve historical context

### Content Guidelines

1. **Clarity and Precision**
   - Use clear, professional language
   - Maintain consistency with existing terminology
   - Preserve legal/policy precision

2. **Cross-References**
   - Maintain clause ID stability (R1, D1, etc.)
   - Update cross-references when making changes
   - Verify all references are valid

3. **Version Control**
   - Document all changes in CHANGELOG.md
   - Use semantic versioning appropriately
   - Preserve historical versions

4. **Extensive Notation**
   - Document decisions and rationale
   - Note history when removing content
   - Explain the "why" behind changes

### Technical Guidelines

1. **Markdown Formatting**
   - Use proper heading hierarchy
   - Maintain consistent formatting
   - Follow existing style patterns

2. **Schema Updates**
   - Maintain backward compatibility where possible
   - Document schema changes
   - Update schema documentation

3. **Testing**
   - Validate cross-references
   - Check clause IDs are unique
   - Verify version numbers are consistent

## Contribution Process

### 1. Identify What You Want to Contribute

- Review existing issues or discussions
- Identify gaps or improvements needed
- Consider how your contribution fits

### 2. Prepare Your Contribution

- Research relevant provisions and frameworks
- Document your rationale
- Ensure consistency with existing content
- Check for conflicts or overlaps

### 3. Submit Your Contribution

**For Small Changes** (typos, clarifications):
- Submit directly via pull request or issue
- Include brief explanation

**For Medium Changes** (new sections, major edits):
- Open an issue first to discuss
- Gather feedback
- Submit pull request with rationale

**For Major Changes** (new versions, structural changes):
- Open discussion issue
- Propose in community forum
- Follow amendment process (Section VII)
- Include comprehensive rationale

### 4. Review Process

- Contributions will be reviewed by maintainers
- AI and human reviewers may participate
- Feedback will be provided constructively
- Revisions may be requested

### 5. Integration

- Accepted contributions will be integrated
- Changes will be documented in CHANGELOG.md
- Contributors will be credited
- Version numbers will be updated as needed

## Amendment Process

For significant changes to the charter itself:

1. **Proposal**: Submit proposal with rationale
2. **Discussion**: Community discussion period (30 days minimum)
3. **Review**: SCB compatibility review per Article 0.5
4. **Vote**: 2/3 Council vote for standard amendments, 3/4 for emergency
5. **Integration**: Update charter and documentation
6. **Documentation**: Update CHANGELOG.md with rationale

See Article 0.5 and Section VII for full amendment procedures.
